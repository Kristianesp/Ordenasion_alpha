# Plan de implementación: resolución de conflictos en preview + release 3.1.0_FIX

## Objetivo
Corregir el flujo crítico de organización para que la vista previa previa a la confirmación no bloquee la operación cuando existan conflictos de nombre/destino y permita al usuario decidir qué hacer de forma clara, profesional y segura.

Además, dejar preparado el build del `.exe` y la publicación de la release `3.1.0_FIX` en GitHub.

## Diagnóstico confirmado

### Problema funcional principal
La vista previa actual detecta conflictos con una lógica demasiado simple:
- `src/gui/preview_dialog.py`
- `PreviewDialog._detect_conflict()` solo hace `destination.exists()`.
- Si existe, marca `Ya existe en destino`.
- No ofrece ninguna política de resolución.
- No simula lo que luego hace realmente el worker.

### Inconsistencia con la ejecución real
El worker sí resuelve conflictos, pero de forma distinta y oculta al usuario:
- `src/core/workers.py`
- En carpetas: renombra con sufijos `_1`, `_2`, etc.
- En archivos: si `check_duplicates` está activo y el archivo es idéntico, lo omite; si no, renombra con sufijos `_1`, `_2`, etc.

Resultado:
- La preview asusta o parece bloquear.
- El usuario no puede decidir.
- La preview no representa el resultado final real.
- El flujo previo a confirmar es poco fiable y ahora mismo inutilizable para conflictos.

### Bug adicional de estado UI
En `src/gui/main_window.py`, `on_analysis_complete()` fuerza:
- `self.organize_btn.setEnabled(True)`
- `self.preview_btn.setEnabled(True)`

Eso contradice `update_selection_count()`, que los habilita solo si hay selección.

Resultado:
- La UI puede quedar habilitada aunque no haya selección real.
- El flujo transmite un estado incorrecto.

### Inconsistencia adicional de preview
`open_selection_preview()` no aplica el filtro de `move_folders_checkbox`, pero `start_organization()` sí lo aplica.

Resultado:
- La preview manual y la preview del flujo real pueden no mostrar lo mismo.

## Recomendación UX

### Opción recomendada
La opción más natural, profesional y sencilla de implementar es:

1. Mantener la `PreviewDialog` como único punto final de confirmación.
2. Añadir dentro del diálogo un bloque visible solo si hay conflictos:
   `Resolución de conflictos`
3. Ofrecer una política global simple para esta ejecución, elegida por el cliente antes de organizar.

Opciones del usuario:
- `Mantener ambos (renombrar el nuevo)`
- `Sobrescribir archivo existente`
- `Omitir elementos en conflicto`

### Política por defecto recomendada
`Mantener ambos (renombrar el nuevo)`

Motivos:
- Es la menos destructiva.
- Es fácil de entender.
- Es la más segura para el usuario.
- Se parece al comportamiento esperado de Windows.
- Requiere cambios acotados y controlables.

### Decisión funcional cerrada
Para `3.1.0_FIX` se ofrecerán las dos opciones principales que pedías para archivos:
- mantener los dos archivos sin sobreescribir, renombrando el nuevo,
- sobrescribir el archivo existente,
- y además una tercera opción práctica: omitir el conflicto.

### Decisión de seguridad importante
Para `3.1.0_FIX`, la opción `Sobrescribir` debe aplicarse solo a conflictos de archivos, no a carpetas completas.

Motivo:
- Sobrescribir carpetas implica merge, reemplazo recursivo o borrado previo.
- Eso complica mucho la semántica, el rollback y el riesgo de pérdida de datos.
- No es un cambio adecuado para un fix rápido y seguro.

Comportamiento recomendado en conflictos de carpeta:
- Siempre usar `Mantener ambos` mediante renombre seguro.
- Si el usuario eligió `Sobrescribir`, mostrar en la preview un microcopy claro:
  `Las carpetas en conflicto se renombrarán automáticamente por seguridad.`

## Diseño funcional propuesto

### Flujo nuevo
1. El usuario analiza y selecciona elementos.
2. Pulsa `Organizar`.
3. Se abre la `PreviewDialog`.
4. Si no hay conflictos:
   - ve la preview final,
   - marca el check de confirmación,
   - organiza.
5. Si hay conflictos:
   - la preview muestra el conflicto,
   - aparece el bloque `Resolución de conflictos`,
   - el usuario elige una política,
   - la tabla se recalcula para mostrar la ruta final real y el estado final real,
   - confirma y organiza.

### Cambio de UX recomendado
Eliminar el `QMessageBox.question()` previo en `start_organization()` y dejar un único paso de confirmación en la `PreviewDialog`.

Motivos:
- Simplifica el flujo.
- Evita doble confirmación.
- La decisión real ya ocurre en la preview.
- Hace el producto más coherente.

## Cambios técnicos propuestos

## 1. Unificar la lógica de conflictos entre preview y worker

### Objetivo
Que preview y ejecución usen exactamente la misma lógica de resolución.

### Implementación recomendada
Crear un helper puro y pequeño reutilizable, por ejemplo:
- `src/core/organization_conflicts.py`

Responsabilidades del helper:
- construir el destino base,
- detectar conflicto de archivo/carpeta,
- generar nombre alternativo con sufijo,
- decidir la acción final según política,
- devolver un resultado estructurado para preview y ejecución.

### API orientativa
Funciones posibles:
- `build_destination_path(...)`
- `compute_conflict_resolution(...)`
- `find_available_name(...)`
- `files_are_identical(...)`

### Motivo de extraer helper
- Evita duplicar reglas entre GUI y worker.
- Evita que vuelva a romperse la coherencia preview/ejecución.
- Permite pruebas unitarias directas.

## 2. Introducir políticas de conflicto explícitas

### Política mínima
Definir un conjunto pequeño y estable:
- `rename`
- `overwrite`
- `skip`

### Regla de nombres recomendada
Cambiar de sufijos `_1`, `_2` a formato más natural:
- `archivo (1).ext`
- `archivo (2).ext`
- `carpeta (1)`

Motivos:
- Más familiar para el usuario.
- Más profesional visualmente.
- Más claro en preview.

## 3. Mejorar `PreviewDialog`

### Archivo principal
- `src/gui/preview_dialog.py`

### Cambios previstos
1. Añadir bloque `Resolución de conflictos`.
2. Mostrarlo solo cuando exista al menos un conflicto.
3. Recalcular `preview_rows` cada vez que cambie la política.
4. Mostrar en `Ruta final` la ruta real resultante según la política seleccionada.
5. Mostrar en `Estado` un resultado semántico más útil, por ejemplo:
   - `Sin conflicto`
   - `Se renombrará a archivo (1).ext`
   - `Se sobrescribirá el archivo existente`
   - `Se omitirá por conflicto`
   - `Duplicado idéntico: se omitirá`
   - `Carpeta existente: se renombrará por seguridad`
6. Mantener el checkbox final de confirmación.
7. No bloquear por existencia de conflictos; solo exigir que la política esté definida y el checkbox esté marcado.

### Datos nuevos a pasar al diálogo
La preview necesita conocer también:
- `check_duplicates`
- política actual por defecto
- si se están moviendo carpetas o solo archivos

## 4. Corregir `main_window.py`

### Archivo principal
- `src/gui/main_window.py`

### Cambios previstos
1. En `on_analysis_complete()`:
   - dejar de forzar `organize_btn` y `preview_btn` a `True`.
   - usar solo `update_selection_count()` como fuente de verdad.

2. Unificar selección real para preview y ejecución:
   - crear un método común para obtener movimientos efectivos,
   - respetando `move_folders_checkbox` tanto en preview manual como en ejecución.

3. En `start_organization()`:
   - eliminar la confirmación redundante previa por `QMessageBox`,
   - abrir directamente la `PreviewDialog` final,
   - recoger del diálogo la política seleccionada,
   - pasar esa política al `OrganizeWorker`.

4. En `open_selection_preview()`:
   - usar exactamente la misma preparación de datos que `start_organization()`.

## 5. Ampliar `OrganizeWorker`

### Archivo principal
- `src/core/workers.py`

### Cambios previstos
Añadir un nuevo parámetro al constructor:
- `conflict_policy: str = "rename"`

### Comportamiento para archivos
#### `rename`
- Si existe conflicto, generar `archivo (1).ext`, `archivo (2).ext`, etc.

#### `overwrite`
- Si el destino existe y el archivo es idéntico, y `check_duplicates` está activo:
  - omitir como duplicado.
- Si el destino existe y no es idéntico:
  - preservar el archivo de destino actual para rollback,
  - mover el nuevo archivo a la ruta original.

#### `skip`
- Si el destino existe:
  - registrar warning o contador,
  - no mover ese elemento.

### Comportamiento para carpetas
Para `3.1.0_FIX`:
- ignorar `overwrite` en carpetas por seguridad,
- renombrar automáticamente con ` (1)`, ` (2)`, etc.
- reflejar esa decisión tanto en preview como en logs.

### Nuevos campos de resumen recomendados
Añadir al summary si resulta útil:
- `overwritten_files`
- `renamed_conflicts`
- `skipped_conflicts`

## 6. Garantizar rollback seguro en sobrescritura de archivos

### Riesgo actual
El `TransactionManager` registra movimientos, pero no modela explícitamente la sustitución de un archivo existente en destino.

### Solución recomendada
Implementar la sobrescritura como secuencia reversible, sin lógica destructiva directa:

1. Mover el archivo existente de destino a una ubicación temporal segura controlada por la transacción.
2. Mover el archivo nuevo al destino final.
3. Registrar ambos movimientos.
4. En rollback, revertir en orden inverso.

### Ventaja
Esto puede apoyarse en las operaciones de tipo `MOVE` ya existentes, minimizando cambios en el motor de rollback.

### Nota
Si durante la implementación aparece una limitación práctica del `TransactionManager`, se podrá añadir una operación específica de reemplazo, pero la primera opción debe ser reutilizar el mecanismo actual para mantener el fix pequeño y seguro.

## 7. Persistencia opcional de preferencia

### Recomendación
No hacer la persistencia obligatoria en la primera iteración del fix.

### Motivo
- El problema urgente es de flujo y ejecución.
- Guardar la última política usada es útil, pero no imprescindible para resolver el bug crítico.
- Si el coste es bajo, se puede añadir en `AppConfig` al final:
  - `organization.conflict_policy`

### Prioridad
- Prioridad alta: resolver flujo.
- Prioridad media: persistir preferencia.

## 8. Plan de pruebas

## Pruebas unitarias
Crear o ampliar tests para:

1. Resolución de nombres:
- genera `archivo (1).ext`, `archivo (2).ext`
- genera `carpeta (1)`

2. Políticas:
- `rename` devuelve ruta alternativa correcta
- `skip` marca omisión
- `overwrite` conserva la ruta final original

3. Duplicados idénticos:
- con `check_duplicates=True`, conflicto idéntico se omite
- con archivo distinto, `overwrite` sustituye

4. Rollback de overwrite:
- el archivo previo se restaura
- el archivo nuevo vuelve a origen si falla la transacción

## Pruebas GUI
Crear o ampliar tests para:

1. `PreviewDialog` con conflictos:
- aparece el bloque de resolución
- el botón final no se bloquea por conflicto
- al cambiar política se actualiza `Ruta final` y `Estado`

2. `PreviewDialog` sin conflictos:
- no aparece el bloque extra
- sigue funcionando como antes

3. `main_window.py`:
- `on_analysis_complete()` no deja botones activos si no hay selección
- preview manual y preview de ejecución usan la misma selección efectiva

## Pruebas manuales
Verificar estos escenarios:

1. Archivo sin conflicto.
2. Archivo con nombre repetido y política `Mantener ambos`.
3. Archivo con nombre repetido y política `Sobrescribir`.
4. Archivo idéntico con `check_duplicates` activo.
5. Archivo con conflicto y política `Omitir`.
6. Carpeta con conflicto de nombre.
7. Organización por fecha con conflicto dentro de subcarpetas `YYYY/MM`.
8. Rollback después de una sobrescritura.
9. Preview manual desde botón `Vista previa`.
10. Flujo completo desde `Organizar`.

## 9. Orden recomendado de implementación

1. Corregir incoherencias de flujo en `main_window.py`.
2. Extraer helper compartido de resolución de conflictos.
3. Adaptar `PreviewDialog` para políticas y recálculo.
4. Adaptar `OrganizeWorker` para respetar la política seleccionada.
5. Implementar sobrescritura reversible para archivos.
6. Añadir pruebas unitarias y GUI.
7. Hacer validación manual completa.
8. Actualizar versión y compilar `.exe`.

## 12. Resultado esperado
Al terminar esta mejora:
- la preview dejará de ser un callejón sin salida,
- el usuario podrá decidir qué hacer cuando exista conflicto,
- la ruta final mostrada será coherente con la ejecución real,
- no habrá sorpresa entre preview y resultado,
- la sobrescritura de archivos será reversible,
- el flujo previo a confirmar volverá a ser usable y confiable.

## Alcance final recomendado para 3.1.0_FIX

### Incluir sí
- política de resolución en preview,
- rename estilo ` (1)`,
- overwrite solo para archivos,
- skip en conflicto,
- coherencia preview/worker,
- fix del estado de botones,
- fix de consistencia entre preview manual y ejecución,
- tests del flujo,
- compilación del `.exe`,
- publicación de release.

### No incluir en este fix
- merge/sobrescritura recursiva de carpetas,
- configuración avanzada por tipo de archivo,
- resolución individual por fila,
- persistencia compleja de preferencias por categoría.

Eso mantiene el fix centrado, seguro y entregable.
