# FUTURO_APP

## Diagnostico ejecutivo

- Si veo la app util y viable como herramienta para ordenar, clasificar y detectar duplicados exactos. Ya tiene una base real: `src/core/workers.py`, `src/core/duplicate_finder.py`, `src/core/transaction_manager.py`, `src/utils/app_config.py` y `src/gui/table_models.py`.
- No la reescribiria desde cero. El mejor camino es refactor incremental con pruebas, manteniendo compatibilidad visual y funcional en cada fase.
- El mayor problema actual no es la falta de funciones, sino la deuda tecnica: monolitos GUI, theming duplicado, logging disperso, estado global defensivo y varias decisiones que ya no escalan bien.
- La caracteristica musical es totalmente viable, pero no debe entrar como "una extension mas". Necesita una capa propia de indexado de audio, calidad, metadatos y duplicados musicales.

## Valor real de la app hoy

- Ordenacion general: util para carpetas mixtas, descargas, discos externos y limpieza basica.
- Duplicados: util para duplicados exactos y sospechosos, pero la presentacion actual mezcla "candidato rapido" con "duplicado confirmado" y eso conviene endurecerlo.
- Visor de discos: aporta valor, pero hoy esta demasiado acoplado a UI y demasiado cargado visualmente.
- Escalabilidad: aceptable para uso medio; todavia no la considero lista para bibliotecas muy grandes ni para una fonoteca seria multi-disco.

## Lo mas obsoleto, fragile o mejorable

- `src/utils/theme_applier.py`: claramente legacy frente a `src/utils/fast_theme_applier.py`.
- `src/gui/main_window.py`: demasiadas responsabilidades en un solo archivo.
- `src/gui/disk_viewer.py`: mezcla UI, cache, historial SMART, analisis y estilos manuales.
- `src/gui/duplicates_dashboard.py`: demasiada logica de pantalla, filtros, acciones y persistencia juntas.
- `src/core/application_state.py`: demasiada logica defensiva/lazy para PyInstaller; funcional, pero costosa de mantener.
- `src/core/worker_manager.py`: o se usa como unico orquestador de workers o sobra complejidad.
- `src/core/hash_cache.py`: el esquema actual no esta preparado para multiples identidades por archivo; ademas usa `file_path` como clave principal aunque tambien guarda `algorithm`.
- `src/core/hash_manager.py`: el hash parcial para archivos gigantes no debe tratarse como verificacion exacta antes de acciones destructivas.
- `src/core/transaction_manager.py`: si `send2trash` no esta instalado, cae a borrado permanente. Para musica y duplicados esto es demasiado arriesgado.
- `src/gui/disk_viewer.py`: abuso de `setStyleSheet()` y uso de `psutil.cpu_percent(interval=1)` en UI, lo que puede bloquear la pantalla.
- `src/gui/main_window.py`: autoanaliza mientras se escribe una ruta; es util para demos, pero peor para uso serio.
- `README.md` y varios comentarios internos ya no describen exactamente la arquitectura real.

## Direccion objetivo

### Arquitectura

- Mantener `src/core`, `src/gui` y `src/utils`, pero introducir capas mas claras.
- Añadir `src/core/models/` para dataclasses de negocio.
- Añadir `src/core/services/` para logica reutilizable.
- Añadir `src/core/workers/` para workers separados.
- Añadir `src/gui/views/` y `src/gui/widgets/` para dividir pantallas grandes.
- Mantener `src/gui/main_window.py` como shell y coordinador ligero.

### Modelo de datos

- Sustituir parte de los `dict` por dataclasses: `MovementItem`, `DuplicateEntry`, `DuplicateGroup`, `DiskHealthSnapshot`, `AudioTrack`, `AudioQuality`.
- Diferenciar siempre entre:
  - candidato rapido,
  - duplicado exacto de fichero,
  - duplicado exacto de audio,
  - duplicado perceptual.

### UI

- Menos emojis en shell principal.
- Menos controles en una sola linea.
- Una sola ruta de theming.
- Mas jerarquia visual y menos ventanas modales.
- Flujos maestro-detalle en discos y duplicados.

### Musica

- Crear subsistema propio de audio sobre SQLite, no sobre `hash_cache.db`.
- Priorizar primero metadatos locales y comparacion tecnica.
- Despues metadatos online y fingerprinting perceptual.

## Reglas para no romper nada durante la implementacion

- Primero pruebas, luego extraccion.
- No mezclar refactor visual grande con refactor de dominio musical en la misma tanda.
- Mantener compatibilidad del `app_config.json` con migraciones pequenas.
- No borrar ficheros de musica automaticamente salvo coincidencias exactas y con ruta segura de recuperacion.
- Introducir nuevas funciones con feature flags o secciones opt-in cuando haga falta.
- Mantener los modelos/contratos viejos durante una fase de transicion si la UI aun depende de ellos.

## Criterios de exito

- `src/gui/main_window.py`, `src/gui/disk_viewer.py` y `src/gui/duplicates_dashboard.py` dejan de ser monolitos.
- No quedan rutas criticas con borrado permanente silencioso.
- Solo existe una ruta oficial de theming.
- La UI principal se ve mas sobria, clara y profesional.
- La app puede indexar y ordenar musica de varios discos por tags y plantillas.
- La app puede mostrar duplicados musicales, comparar calidad y sugerir la mejor copia.
- La app puede enriquecer metadatos online sin hacer cambios destructivos a ciegas.

## Conclusiones finales

- La herramienta tiene futuro real. Ya es util para ordenacion general y duplicados exactos.
- El salto de calidad no depende de meter mas funciones encima, sino de refactorizar bien el nucleo y la GUI.
- La parte musical merece convertirse en una linea principal del producto, no en una opcion secundaria.
- Si se sigue este orden de fases, puedes llegar a una app muy potente para ordenar archivos y especialmente para poner orden serio en una biblioteca musical grande sin perder las funciones ya existentes.
